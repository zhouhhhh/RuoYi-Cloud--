# RuoYi-Cloud 学习路线设计

## 目标

本项目采用项目驱动方式学习 RuoYi-Cloud。学习者每周投入约 5 至 10
小时，通过逐步开发一套客户关系管理系统，最终能够独立完成微服务开发、
服务治理、测试、部署和故障排查。

学习路线不以复制完整代码为目标。每个阶段都必须经过概念理解、自主实现、
运行验证、代码审查和复盘总结。

## 技术基线

- 后端：Java 17、Spring Boot 3.5、Spring Cloud 2025、Spring Cloud
  Alibaba、MyBatis。
- 前端：Vue 3、Vite、Element Plus。
- 基础设施：MySQL、Redis、Nacos、Gateway、Sentinel、Seata、Docker
  Compose。
- 主后端目录：`ruoyi-cloud-learning/RuoYi-Cloud-springboot3`。
- 主前端目录：`ruoyi-cloud-learning/RuoYi-Cloud-Vue3-master`。
- 后端仓库中无关的 Vue2 `ruoyi-ui` 目录应删除，避免维护两个前端副本。

## 路线方案

采用渐进式拆分：

1. 先建立可重复启动和排错的运行基线。
2. 在 `ruoyi-system` 中完成客户业务闭环。
3. 加入权限、日志、文件和任务等 RuoYi 平台能力。
4. 将客户业务迁移到独立 `ruoyi-customer` 微服务。
5. 增加 `ruoyi-contract`，形成真实的跨服务调用和事务场景。
6. 完成缓存、幂等、限流、熔断、分布式事务、监控和容器化交付。

这样安排可以先隔离业务错误与基础设施错误，再通过迁移理解微服务边界及其
成本。

## 学习里程碑

1. 环境基线：启动基础设施、核心后端服务和 Vue3 前端，记录端口与顺序。
2. 请求链路：跟踪登录和查询请求经过前端、Gateway、Auth、System、
   Redis 和 MySQL 的过程。
3. 客户档案：设计 `crm_customer`，通过代码生成建立 CRUD，并修正生成结果。
4. 业务校验：实现手机号唯一、状态流转、参数校验和统一异常处理。
5. 联系人管理：开发 `crm_contact`，掌握一对多关系和关联查询。
6. 跟进记录：开发 `crm_follow_record`，实现详情时间线和本地事务。
7. 权限体系：配置菜单、按钮、角色以及按部门或负责人划分的数据权限。
8. 平台组件：完成 Excel 导入导出、操作日志、附件上传和文件访问。
9. 定时任务：识别长期未跟进客户，更新状态并记录任务日志。
10. 服务拆分：迁移到 `ruoyi-customer`，配置 Maven、Nacos 和 Gateway。
11. 服务调用：使用 OpenFeign 获取负责人及部门信息，并处理超时和异常。
12. 合同服务：建立轻量 `ruoyi-contract`，形成两个业务服务间的调用。
13. 缓存与幂等：缓存客户摘要，处理缓存一致性、穿透和重复提交。
14. 流量治理：配置 Gateway 限流和 Sentinel 熔断、降级，执行故障演练。
15. 分布式事务：创建合同时更新客户成交状态，用 Seata 验证提交和回滚。
16. 交付验收：增加核心测试、监控、Docker Compose、配置隔离和部署文档。

## 任务结构

`AGENTS.md` 中的每个里程碑都应包含：

- 状态复选框。
- 开发内容。
- 必须理解的概念。
- 完成标准。
- 验证证据。

一次只推进一个里程碑。上一个里程碑没有通过验收时，不进入依赖它的后续
任务。

## 数据流与服务边界

基础阶段的请求链路为：

`Vue3 -> Gateway -> Auth/System -> MyBatis -> MySQL`

拆分后的业务链路为：

`Vue3 -> Gateway -> Customer/Contract -> OpenFeign -> 其他服务`

`ruoyi-customer` 负责客户、联系人、跟进记录和附件元数据；
`ruoyi-contract` 负责合同及签约流程；`ruoyi-system` 保持用户、部门、
角色和菜单等平台职责。服务之间通过接口协作，不直接读取对方数据库。

## 异常处理与排错

故障必须按链路分层定位：

1. 前端页面、控制台和 Network。
2. Gateway 路由、鉴权和限流。
3. Auth 与权限上下文。
4. Customer、Contract、System 等业务服务日志。
5. Nacos 注册与配置。
6. Redis 缓存和令牌状态。
7. MySQL 连接、SQL 与事务。
8. Docker 容器、端口和网络。

学习记录需要写明现象、所属层、证据、根因和修复结果，不能只记录最终答案。

## 测试与验收

任务完成至少提供以下证据：

- Maven 或前端构建结果。
- 核心单元测试或集成测试结果。
- 正常请求及关键失败请求的响应。
- 数据库或缓存状态。
- 关键服务日志。
- 学习者对请求链路和实现原因的简短说明。

分布式阶段必须包含超时、服务不可用、重复请求和事务回滚等失败场景。

## AI 协作边界

- AI 默认遵守 `docs/ai-tutor-mode.md`，先解释概念，再拆分任务。
- AI 默认提供伪代码、接口签名、关键片段和调试步骤，不直接交付完整功能。
- 学习者完成代码后，AI 按 `docs/ai-review-mode.md` 审查正确性、风险和测试。
- 只有学习者明确要求“直接实现”或“给完整代码”时，AI 才代为修改完整功能。
- AI 不得把任务标记为完成，除非已有可复核的验证证据。
